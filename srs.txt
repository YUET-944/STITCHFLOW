A.4 Strategic Positioning Statement
Why StitchFlow Wins
Every existing platform treats the tailoring relationship as a tailor-centric data problem. StitchFlow reframes it as a bilateral trust problem. The tailor needs an intelligent business OS; the client needs data ownership and transparent production visibility. No existing platform provides both simultaneously. StitchFlow is the first dual-actor platform in this market: a fully-featured Tailor Business App fused with a Client Transparency Portal, sharing one backend and one unified ID system. This is not a feature gap — it is a fundamental architectural difference.
1. Introduction
1.1 Purpose
This document constitutes the authoritative Software Requirements Specification (SRS) and Software Design Description (SDD) for StitchFlow — a dual-facing cross-platform tailoring ecosystem. It is produced as a sprint-ready technical directive for the engineering team, covering all functional requirements, data architecture, API contracts, permission logic, and UI directives required to build, test, and deploy version 1.0.
1.2 Scope
StitchFlow solves three structurally-identified innovation problems simultaneously:
The Black Box Problem: Replacing opaque status updates with a client-verified, granular production checklist at the garment-component level.
The Measurement Portability Gap: Building a client-owned Measurement Vault that persists across tailors and is searchable by a unique ID.
The Manual POS Trap: Unifying invoicing and production workflow into a single POS action that generates both a billing invoice and a digital stitching ticket.

2. Core Innovation Specifications
2.1 Innovation I — Solving the Black Box Problem
2.1.1 Problem Statement
In all existing platforms, order status is a single-field enum (Pending / Ready / Delivered). The client never knows whether 'stitching' means their suit jacket is being cut, the lining is being attached, or buttons are being stitched. This opacity generates repeated phone calls, erodes trust, and leaves tailors managing customer anxiety instead of garments.
The Black Box in Numbers
Per competitor review analysis: tailors in India receive an average of 3–5 status inquiry calls per active order per week. At 20 active orders, that is 60–100 unnecessary calls weekly — dead time that StitchFlow eliminates entirely.
2.1.2 StitchFlow Solution — Granular Requirements Checklist
StitchFlow replaces the single-field status with a three-layer transparency architecture:
Order-Level Status: PENDING / CONFIRMED / IN_PRODUCTION / READY / DELIVERED — visible to both tailor and client.
Garment-Level Stage: Per-garment production checklist with the following standard stages:
Measurement Confirmed
Fabric Cut
Interlining / Padding Attached
Main Body Stitched
Collar / Lapel Attached
Sleeves Set
Pockets Sewn
Lining Installed
Buttons / Closures Attached
Pressing & Finishing
Quality Check Passed
Requirement Verification: At order creation, a pre-production checklist is generated for the client to verify preferences before production begins (Neck type, Pocket style, Sleeve length, Lining colour, Collar style). This becomes part of the Stitching Ticket.
Client Verification Flow
Before the Tailor advances any order past CONFIRMED, the Client receives a 'Verify Requirements' prompt on their app. They confirm or request changes to: Neck Type, Pocket Style, Sleeve Treatment, Lining, Button Colour. Only on client confirmation does the system allow the Tailor to advance to IN_PRODUCTION. This single feature eliminates the most common cause of rework: miscommunicated preferences.
2.2 Innovation II — Solving the Measurement Portability Gap
2.2.1 Problem Statement
In all existing platforms — TailorMate, Darzee, Atelierware, DarzeeBook — measurements are stored in the tailor's account, not the client's. If a client changes tailors, their measurement history disappears. The client has no proof of their own body data. This creates a dependency relationship that disincentivises client loyalty and makes data portability impossible.
2.2.2 StitchFlow Solution — The Client-Owned Measurement Vault
The Vault is the Client's permanent, cross-tailor measurement record. It persists even if the client works with zero active tailors.
Any Tailor with an active or confirmed order can write to a Client's Vault. Multiple tailors may have contributed measurement records over time.
The Client can view all measurement versions, all contributing tailors, and all timestamps — full audit trail.
The Client may export their Vault as a PDF or share via a one-time link.
Only the Client's assigned Primary Tailor (the tailor on the most recent confirmed order) may edit the current measurement record.
Vault data is encrypted at rest (AES-256). The encryption key is derived from the Client's UID — not stored with the tailor.
Anonymous ID Search — Vault Access Without Login
A non-registered user can enter a Client Readable ID (e.g. CL-Lahore-042) on the StitchFlow public web portal and retrieve: the client's first name, their current active order status, and the production checklist. They CANNOT access raw measurement data without authentication. This enables family members, friends, or gift-givers to check delivery status without accounts.
2.3 Innovation III — Solving the Manual POS Trap
2.3.1 Problem Statement
Tailors today use separate systems for billing (manual book or basic invoicing app) and production tracking (verbal or paper-based stitching slips). There is no platform that connects a financial transaction to a garment production workflow in a single action. RepairDesk comes closest for repair shops but is not tailoring-native — it has no concept of garment components, measurement fields, or stitching stages.
2.3.2 StitchFlow Solution — Unified POS with Auto-Generated Stitching Ticket
When a Tailor completes a booking confirmation and initiates the order in StitchFlow's Tailor POS, one button press generates two documents simultaneously:
Digital Invoice: Itemised bill with garment type, base price, advance received, balance due, and estimated delivery date. Shareable via WhatsApp or in-app message.
Digital Stitching Ticket: A production card containing client name, readable ID, measurements, garment-level requirements checklist, assigned Staff Master, delivery deadline, and a QR code that links to the live production status. Printable on thermal (58mm) or A4.
One-Click Impact
The POS → Stitching Ticket flow eliminates an estimated 8–12 minutes of manual data re-entry per order. For a tailor managing 30 orders per month, this saves 4–6 hours monthly and eliminates the most common source of measurement transcription errors.
3. Functional Requirements — Master List
3.1 Tailor POS & Business Application
3.1.1 Onboarding & Profile
FR-T-001: Tailors shall register via the Dual-Entry Gateway selecting 'Join as Tailor'. Required fields: Business Name, Phone (OTP-verified, E.164), Location (Google Maps Places API, stored as GeoPoint), Specialization Tags (multi-select: Suits, Shalwar Kameez, Sherwani, Bridal, Alterations, Ladies, Gents, Kids), and per-suit base pricing range.
FR-T-002: Tailors shall receive a system-generated Readable ID in format TR-{City}-{Sequence} (e.g. TR-Mardan-001).
FR-T-003: Tailor accounts require admin verification before the capacity gate and online booking become active. An 'Unverified' badge displays on profile.
3.1.2 Unified POS & Billing
FR-T-010: The POS module shall support: garment type selection, base price entry, advance payment capture (cash/bank transfer/mobile money), automatic balance calculation, and delivery date setting.
FR-T-011: On POS confirmation, the system shall simultaneously generate a digital Invoice (PDF/WhatsApp-shareable) and a digital Stitching Ticket (printable QR-coded card).
FR-T-012: The Stitching Ticket shall contain: Client Name, Client Readable ID, all measurement fields for the relevant garment type, the requirements checklist, assigned Staff Master, delivery date, and a QR code linking to the live order tracker.
FR-T-013: The invoice shall display: Order ID, Garment type, Price, Advance paid, Balance due, Estimated delivery date, and Tailor's business address.
FR-T-014: Payment tracking shall maintain three states per order: ADVANCE_RECEIVED, BALANCE_PENDING, FULLY_PAID.
3.1.3 Staff Assignment
FR-T-020: Tailors shall be able to add Staff Profiles (Name, Specialty, Phone) to their workspace.
FR-T-021: Each garment within an order shall be assignable to a specific Staff Master. This assignment appears on the Stitching Ticket and in the staff's personal dashboard view.
FR-T-022: A Staff Performance view shall display: total garments assigned, completed on-time, completed late, and average turnaround time per staff member.
3.1.4 Capacity Management & Booking Approval
FR-T-030: Tailors shall configure Max Active Orders (integer 1–99). The system enforces this ceiling via a serialised database transaction (no race conditions).
FR-T-031: When current_active_orders >= max_active_orders, the online booking portal shall display 'Currently Full — Request Waitlist' and disable new booking submission.
FR-T-032: Incoming online booking requests appear in the Booking Queue with PENDING status. Tailors may Approve or Reject each request.
FR-T-033: Rejection requires a reason code: FULLY_BOOKED | SPECIALIZATION_MISMATCH | TIMELINE_UNAVAILABLE | CLIENT_REQUEST_UNCLEAR | OTHER. An optional free-text note is available.
FR-T-034: On Approval, the system: (a) increments current_active_orders, (b) creates a tailor_client_link if one does not exist, (c) creates a Firebase Realtime Database chat channel, (d) sends FCM push and in-app notification to the Client.
3.1.5 Portfolio Gallery
FR-T-040: Tailors shall upload portfolio images (JPEG/PNG/HEIC, max 10MB) via a Cloudinary-signed upload flow. Images are auto-converted to WebP.
FR-T-041: Each portfolio image shall support Style Tags (multi-select from predefined + custom: Formal, Casual, Bridal, Sherwani, Three-Piece, Two-Piece, Embroidered, Printed, etc.).
FR-T-042: The portfolio gallery renders as a grid (3-column on mobile, 4-column on web) with style tag filter chips.
FR-T-043: Maximum 50 images per tailor profile. Drag-to-reorder supported. Cover image selectable.
3.2 Client Application & Portal
3.2.1 Client Onboarding & Vault
FR-C-001: Clients register via 'Join as Client' gateway. Required: Full Name, Phone (OTP-verified). Optional: Profile Photo.
FR-C-002: A Client Readable ID is generated: CL-{City}-{Sequence} (e.g. CL-Karachi-042).
FR-C-003: The Measurement Vault displays all measurement records linked to the client's ID — sorted by date, grouped by contributing tailor. Each record shows all fields, the recording tailor's name, and the timestamp.
FR-C-004: Clients cannot edit any measurement field directly. An 'Update Request' button sends a message to their primary tailor.
FR-C-005: Clients can export their full Vault as a PDF containing all historical measurement versions.
3.2.2 Anonymous ID-Based Order Tracking
FR-C-010: A public web portal (no login required) shall accept any Client Readable ID in a search bar and return: Client first name, current active order status, garment type, production checklist with completed stages highlighted, and expected delivery date.
FR-C-011: Raw measurement data, full contact information, and financial details shall NOT be accessible without authentication.
FR-C-012: The Client app's 'Track' tab shall display the same checklist with real-time Socket.io/Supabase Realtime updates — no pull-to-refresh required.
3.2.3 Requirements Verification Checklist
FR-C-020: Upon booking confirmation, the Client receives a 'Verify Your Requirements' prompt presenting garment-specific preference fields for their review and approval.
FR-C-021: The Tailor's POS is locked from advancing to IN_PRODUCTION until the Client has submitted the requirements verification. A countdown timer (24 hours) auto-approves with default values if the client does not respond.
FR-C-022: The verified requirements are frozen into the Stitching Ticket and are read-only post-approval.
3.2.4 Booking Flow
FR-C-030: Clients may initiate a Suite Slot booking from any verified Tailor's public profile. The booking form captures: Garment Type, Preferred Date Range, Special Instructions (max 500 chars), and up to 5 reference/fabric photos.
FR-C-031: The booking form shall display the Tailor's current capacity indicator (e.g., '7 of 10 slots filled') and estimated queue position.
FR-C-032: On submission, the client receives a booking confirmation notification and a 'PENDING — Awaiting Tailor Approval' status on their Orders screen.
3.2.5 Communication
FR-C-040: Verified phone numbers of assigned Tailors are visible to Clients with confirmed orders. On mobile, a 'Call' button triggers the native dialler via url_launcher (tel: URI). On web, the number is displayed with a copy button.
FR-C-041: An in-app chat channel is created upon booking confirmation. Supports TEXT messages, IMAGE attachments (fabric/reference photos), and SYSTEM messages (auto-generated on stage transitions).
FR-C-042: Chat messages are delivered via Firebase Realtime Database at sub-200ms latency. FCM push is dispatched when the recipient is backgrounded.
FR-C-043: A 'Saved Tailors' list allows clients to bookmark preferred tailor profiles for quick re-booking.

6. Database Schema Blueprint
6.1 Entity Relationship Overview
The schema is designed around a Client-centric data model where the Client ID is the primary axis. The One-to-Many relationships are:
1 Client → Many Orders (across multiple tailors over their lifetime)
1 Client → Many Measurement Records (versioned, from multiple tailors)
1 Tailor → Many Orders (their active workload)
1 Order → 1 Stitching Ticket, 1 Invoice, 1 Chat Channel, Many Stage History Records
1 Order → Many Garment Items (a single order can contain multiple garments)
1 Garment Item → 1 Staff Master assignment, 1 Requirements Checklist, Many Stage Updates
Primary Key Convention
Users: Firebase Auth UID (28-char alphanumeric, system-generated)
Readable IDs: TR-{City}-{ZeroPaddedSeq} / CL-{City}-{ZeroPaddedSeq} (human-searchable, unique)
Orders: SF-O-{T4}-{C4}-{EpochMs}  |  Measurements: SF-M-{T4}-{C4}-{EpochMs}
Stitching Tickets: SF-ST-{OrderId}  |  Invoices: SF-INV-{TailorShortId}-{YY}-{Seq}

7. Data Flow & Permission Logic
7.1 One-to-Many: Client ID and Orders
The Client Readable ID (CL-{City}-{Seq}) serves as the universal lookup key across the entire platform:
One Client ID → Many Orders: A single client may have orders with multiple tailors simultaneously or sequentially. There is no restriction on order count per client.
One Client ID → Many Measurements: Measurements from multiple tailors are all linked to the same Client ID, forming the Vault's historical record.
One Client ID → One Anonymous Search Result: The public portal always returns the most recent active order for a given Client ID.

7.3 Measurement Write Access — Triple Gate
Security Gate Implementation
Gate 1 — JWT Role Check (API Middleware): Request must carry JWT with role='TAILOR'. HTTP 403 if role mismatch.

Gate 2 — Link Existence Check (API Service Layer): SELECT 1 FROM tailor_client_links WHERE tailor_id = $uid AND client_id = $targetClientId AND is_active = true. HTTP 403 with code LINK_NOT_FOUND if absent.

Gate 3 — Supabase RLS Policy (Database Layer): CREATE POLICY measurement_write ON measurements FOR INSERT WITH CHECK (tailor_id = auth.uid() AND EXISTS (SELECT 1 FROM tailor_client_links WHERE tailor_id = auth.uid() AND client_id = measurements.client_id AND is_active = true));

All three gates must pass. A compromised API layer cannot bypass the RLS policy at the database level.
7.4 Capacity Gate — Concurrency-Safe Implementation
Serialised Transaction Logic (Node.js Booking Service)
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
  SELECT current_active_orders, max_active_orders, availability_status
  FROM tailor_profiles WHERE tailor_id = $tailorId FOR UPDATE;  -- row lock
  IF current_active_orders >= max_active_orders:
    ROLLBACK;
    THROW CapacityExceededException { code: 'CAPACITY_EXCEEDED', http: 409 };
  IF availability_status != 'ACTIVE':
    ROLLBACK;
    THROW UnavailableException { code: 'TAILOR_UNAVAILABLE', http: 409 };
  INSERT INTO orders (...) VALUES (...) RETURNING id;
COMMIT;

The FOR UPDATE lock prevents race conditions when two clients submit simultaneously. The serialised isolation level handles edge cases under high concurrency.
